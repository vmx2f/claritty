## 5) Code Style Guidelines

### TypeScript and typing

- TS strict mode is enabled; keep code type-safe.
- Prefer explicit domain types/interfaces for public function boundaries.
- Avoid `any`; use `unknown` + narrowing when needed.
- Keep null/undefined handling explicit (avoid unsafe non-null assertions).
- Reuse Convex generated types (`Id<"table">`, `Doc`, etc.) where applicable.

### Imports

- Use `@/*` alias for `src/*` imports when practical.
- Keep imports grouped and stable:
  1) external packages,
  2) internal aliases/relative modules,
  3) type-only imports where useful.
- Remove unused imports; keep each file's import surface minimal.

### Formatting and layout

- Follow existing style in the touched file (quote style/spacing are mixed today).
- Keep functions/components small and readable.
- Prefer early returns for guard clauses.
- Do not rewrap/reorder unrelated JSX/Tailwind classes just for style.
- Add comments only when logic is non-obvious.

### Naming

- React components: PascalCase (`DashboardLayout`).
- Hooks: `useXxx` naming.
- Variables/functions: camelCase.
- Constants: UPPER_SNAKE_CASE for true constants (e.g., currency constants).
- Convex function exports: descriptive camelCase (`createOrganization`).
- Route segment folders follow Next.js conventions (`[locale]`, `(group)`).

### React/Next.js patterns

- Prefer Server Components by default; add `"use client"` only when required.
- Keep client components focused on UI state/effects.
- Use `next/navigation` APIs in client components where navigation is needed.
- Keep providers centralized (`src/app/[locale]/providers.tsx`).

### Tailwind/UI

- Use Tailwind utility classes as primary styling approach.
- Reuse existing visual tokens/classes/patterns before adding new ones.
- Keep accessibility in mind (labels, focus styles, button semantics).

### i18n

- User-facing strings should be translatable where the feature already uses i18n.
- Follow existing `next-intl` setup and locale routing conventions.
- Update corresponding `messages/*.json` entries for new copy.

### Convex backend patterns

- Validate args with `v` validators in every query/mutation/action.
- Perform auth checks early (`ctx.auth.getUserIdentity()`) for protected ops.
- Return consistent shapes; avoid overly broad `v.any()` unless necessary.
- Keep permission checks close to data mutations.
- Update timestamps (`createdAt`/`updatedAt`) consistently when mutating records.

### Error handling

- Fail fast on authorization/validation issues.
- Throw clear errors in server/Convex code (`"Unauthorized"`, permission errors).
- In client UI, catch async errors and expose friendly messages/state.
- Do not swallow errors silently.

### Environment/config

- Add/modify env vars through `src/env/client.ts` or `src/env/server.ts` schemas.
- Keep `NEXT_PUBLIC_*` vars in client env only.
- Avoid reading undeclared env vars directly in app code.
